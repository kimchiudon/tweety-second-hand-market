# Tweety Second-hand Shopping Platform 개발 보고서

## 1. 프로젝트 개요

본 프로젝트는 중고 상품을 등록·검색·구매하고 사용자끼리 소통할 수 있는 교육용 웹 플랫폼이다. 수업 자료의 7개 요구사항을 모두 구현하면서, 기능 구현 이후에 보안을 덧붙이는 방식이 아니라 요구사항·설계 단계부터 오용 사례를 함께 정의했다. 구현은 Python 3.11+, SQLite와 이미지 전체 검증·재인코딩을 위한 Pillow를 사용했다.

## 2. 요구사항 분석

### 2.1 기능 요구사항

수업 자료 35쪽의 최소 요구사항을 기준으로 다음과 같이 대조했다.

| 최소 요구사항 | 구현 근거 | 결과 |
|---|---|---|
| 플랫폼 가입 | 회원가입·로그인, 고유 아이디·닉네임 | 충족 |
| 상품 등록·조회 | 최대 10장 사진, 목록·상세, 본인 상품 관리 | 충족 |
| 사용자 간 소통 | 상품 상세에서 시작하는 상품별 1:1 채팅 | 충족 |
| 악성 사용자·상품 차단 | 신고 누적 제한, 사용자 차단, 관리자 조치 | 충족 |
| 유저 간 송금 | 구매자→판매자 교육용 포인트 이전과 불변 원장 | 충족 |
| 상품 검색 | 검색어와 한 글자 이상 겹치는 상품명 검색, SQL 매개변수 처리 | 충족 |
| 관리자의 전체 요소 관리 | 사용자·신고·차단·계정·신고 상품 관리 | 충족 |

25~26쪽에는 전체 채팅과 1:1 채팅이 함께 설계 예시로 나오지만, 35쪽 최소 요구사항은 “사용자 간 소통”으로 표현된다. 본 구현은 사용자가 요청한 상품 맥락 중심 1:1 채팅으로 이를 충족하며 전체 공개 채팅은 의도적으로 제외했다.

1. 사용자 가입, 고유 닉네임, 로그인, 프로필·비밀번호 변경
2. 사진 업로드 상품 등록, 전체/상세 조회, 검색, 판매자 수정·삭제
3. 상품별 판매자-문의자 1:1 메시지
4. 사용자 차단·해제, 사용자·상품 신고, 실제 참여자만 가능한 채팅 신고와 누적 신고 자동 제한
5. 전용 확인 화면을 거친 구매자에서 판매자로의 교육용 포인트 송금
6. 관리자의 신고 처리, 차단 관계 관리, 계정 활동 정지·해제와 신고 상품 삭제
7. 내 판매·구매 내역과 잔액 확인

### 2.2 비기능 요구사항

- Python 3.11 이상과 프로젝트에 고정된 호환 범위의 Pillow로 재현 가능하게 설치되어야 한다.
- 비밀번호, 세션, 개인 메시지, 거래 잔액을 보호해야 한다.
- 일반 사용자가 다른 사용자의 객체나 관리자 기능에 접근할 수 없어야 한다.
- 동시 구매에서도 한 상품은 한 번만 판매되고 포인트 총량이 보존되어야 한다.
- PC와 모바일에서 핵심 기능을 사용할 수 있어야 한다.

### 2.3 오용 사례

- 공격자가 상품 ID를 바꾸어 다른 판매자의 상품을 수정·삭제한다.
- 로그인된 사용자가 악성 링크를 눌러 의도치 않은 구매 요청을 보낸다.
- 상품명·채팅에 스크립트를 저장해 다른 사용자의 브라우저에서 실행한다.
- 두 구매자가 동시에 마지막 상품을 구매해 판매가 중복 처리된다.
- 한 사용자가 신고를 반복해 정상 상품이나 계정을 자동 차단한다.
- 일반 사용자가 관리자 URL을 직접 입력해 관리 기능을 실행한다.

## 3. 시스템 설계

### 3.1 구성

브라우저는 WSGI 애플리케이션에 HTTP 요청을 보낸다. 라우터가 세션·CSRF 검증 후 기능별 처리로 전달하며, 모든 영속 데이터는 SQLite에 저장한다. 동적 HTML은 서버에서 생성하고 모든 사용자 입력을 escape한다. 정적 CSS와 로컬 이미지 외 외부 리소스는 사용하지 않는다.

### 3.2 데이터 모델

- `users`: 아이디, 비밀번호 해시, 소개글, 잔액, 역할, 계정 상태
- `sessions`: 토큰 해시, 사용자, CSRF 토큰, 만료 시각
- `products`: 판매자·구매자, 상품 정보, 판매·노출 상태
- `messages`: 상품, 발신자, 수신자, 1:1 대화 본문
- `reports`: 신고자, 대상 유형·ID, 사유, 처리 상태와 중복 방지 제약
- `user_blocks`: 차단자, 차단 대상, 생성 시각과 중복·자기 차단 방지 제약
- `product_images`·`message_images`: 상품·메시지별 최대 10장 사진과 표시 순서
- `transfers`: 송금자, 수신자, 상품, 금액의 불변 원장
- `login_attempts`, `audit_log`: 악용 방지와 보안 이벤트 추적

### 3.3 권한 모델

비로그인 사용자는 공개 상품만 조회한다. 일반 사용자는 본인 프로필·상품과 자신이 참여한 1:1 메시지만 관리한다. 활동 정지 사용자는 로그인해 본인 내역과 기존 채팅을 읽을 수 있지만 상품 등록·수정·구매와 메시지 전송은 할 수 없고, 정지 판매자의 상품은 거래 목록에서 제외된다. 상품 구매는 본인 상품이 아니고 노출·판매 중이며 판매자·구매자 계정이 모두 활성 상태이고 잔액이 충분할 때만 가능하다. 관리 기능은 매 요청마다 세션의 관리자 역할을 확인한다. 관리자는 사용자 비밀번호를 조회하거나 임의로 설정할 수 없다.

## 4. 구현

`tiny_market/app.py`는 라우팅, 인증/인가, 상품·채팅·신고·거래·관리자 로직을 담당한다. `security.py`는 PBKDF2, 토큰 생성, CSRF 비교, 입력 파싱을 캡슐화한다. `db.py`는 SQLite 연결과 원자적 트랜잭션 경계를 제공한다. `views.py`는 HTML 이스케이프가 적용된 화면을 생성하고 `static/style.css`가 반응형 UI를 제공한다.

플랫폼 이름은 `Tweety`로 정하고, “Tiny보다 더 작은 장터”라는 콘셉트를 노란 오리, 미니 장바구니, 가격표로 표현했다. 마스코트는 수업 프로젝트용으로 새로 만든 오리지널 디자인이며 동명의 기존 캐릭터 이미지를 사용하지 않았다. 크림·노랑·짙은 청록·주황 색상과 둥근 카드 UI를 적용하고, 상품 사진에는 `object-fit: contain`을 사용해 원본이 잘리지 않도록 했다.

포인트 송금 확인 화면은 상품, 받는 사람, 보유 포인트, 송금액, 송금 후 잔액을 명시한다. 이는 외부 PG나 실제 금융 결제가 아닌 교육용 내부 포인트 송금이다. 확정 요청에서는 `BEGIN IMMEDIATE`로 쓰기 잠금을 얻은 뒤 구매자 잔액 조건부 차감, 상품 판매 상태 조건부 변경, 판매자 적립, 원장 기록을 같은 트랜잭션에서 수행한다. 조건을 만족하지 않으면 변경 없이 롤백/종료하므로 중복 구매와 음수 잔액을 막는다.

## 5. 보안 약점 발견 및 변경

초기 기능 구상과 구현 검토에서 세션 고정, CSRF, SQL 삽입, 저장형 XSS, IDOR, 무차별 로그인, 계정 존재 시간 추측, 신고 반복, 이중 구매, 로그 개인정보, 관리자 권한 우회, 이미지 형식 위장·메타데이터·대용량 요청, 공개 URL을 통한 채팅 사진 노출 가능성을 확인했다. 각각 세션 회전, CSRF 토큰, 매개변수 SQL, 출력 이스케이프+CSP, 객체 소유권 재검사, 계정별·IP별 속도 제한과 동일 비용 더미 해시, 신고자별 UNIQUE, 즉시 트랜잭션, privacy hash, 서버 역할 검사로 변경했다. 사진은 WebP를 제외한 PNG/JPEG만 확장자·MIME·시그니처와 전체 디코딩으로 확인한 뒤 재인코딩하여 EXIF·후행 데이터를 제거하며, 채팅 사진은 대화 참가자에게만 제공한다. 상세 변경 전후와 테스트 근거는 `SECURITY.md`에 정리했다.

## 6. 체크리스트 및 테스팅

기능·보안 체크리스트를 먼저 만들고 정상/비정상/권한/경계/동시성 관점의 자동 테스트 28개를 작성했다. 최종 결과는 28개 모두 성공했다. 닉네임 가입·중복, 상품명 한 글자 이상 일치 검색과 SQL 삽입 방어, 상품·채팅 사진 최대 10장과 11장 거부, WebP·위장 파일 거부, 재인코딩 후 메타데이터 제거, 채팅 사진의 참가자 전용 조회, 계정별·IP별 로그인 제한, 채팅 알림·읽음 처리, 실제 대화 참여자만 가능한 채팅 신고와 관리자 화면의 상품 맥락 확인, 상품별 대화 격리, 송금 확인 화면, 차단 후 상품·채팅·구매 제한, 활동 정지 계정의 로그인·조회 허용과 거래·메시지 제한, 관리자 신고·차단·계정 정지 관리, 신고상품 삭제, 배포 상태 확인과 관리자 최초 생성의 멱등성을 검증했다. 실제 브라우저에서는 회원가입 후 세션이 정상 회전되는지, 상품 등록 후 상세 페이지가 표시되는지, CSP 환경에서 화면이 정상인지 확인했다. 390px 모바일 점검에서 가로 넘침을 발견해 메뉴 줄바꿈과 상세 카드 폭 제한을 추가하고 재검증했다.

## 7. 유지보수

Render 배포 설정은 HTTPS 종단, 운영 Gunicorn, 상태 확인, 단일 인스턴스와 영구 디스크를 사용한다. 운영 중에는 DB 백업·복원과 로그 모니터링을 수행해야 한다. 인증·거래·관리자 코드가 바뀔 때마다 전체 회귀 테스트를 수행하고, DB 구조 변경에는 명시적 마이그레이션을 도입한다. 신고 임계값은 오탐과 미탐을 분석해 조정하며, Python 보안 업데이트를 정기 적용한다.

## 8. AI 도구 활용 및 검증

Codex를 수업 PDF 요구사항 추출, 위협 모델링, 구현 초안, 공격 테스트 작성, 실제 브라우저 검수, 문서와 보고서 제작에 적극 활용했다. AI가 제안한 코드는 그대로 신뢰하지 않고 자동 테스트, DB 불변조건 확인, 데스크톱·모바일 브라우저 점검으로 검증했다. 특히 익명 세션과 로그인 세션 쿠키가 동시에 내려갈 수 있는 문제와 모바일 가로 넘침은 검증 과정에서 발견해 수정했다.

## 9. 결과 및 한계

수업 자료 35쪽의 최소 요구사항 7개를 모두 충족하고 보안 요구를 자동 테스트로 재현했다. 25~26쪽의 전체 채팅은 설계 예시로 판단했으며, 개인정보 노출 면적을 줄이고 상품 맥락을 보존하려는 요구에 따라 상품별 1:1 채팅만 제공한다. 포인트는 실제 금융 결제가 아니며, 채팅 알림은 10초 주기이고 종단간 암호화되지 않는다. 현재 서버는 로컬 수업용이므로 인터넷 공개 배포 시 README의 운영 권고를 적용해야 한다.

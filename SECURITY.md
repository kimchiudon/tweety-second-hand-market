# Security Policy

## 위협 모델

보호 대상은 계정 인증정보, 서버 세션, 개인 메시지, 상품/거래 무결성, 포인트 잔액입니다. 공격자는 비로그인 외부인, 권한이 낮은 일반 사용자, 동시 구매를 시도하는 사용자로 가정했습니다. 서버 운영체제나 관리자 단말이 이미 장악된 경우는 범위 밖입니다.

## 개발 중 확인하고 변경한 보안 약점

| 확인한 약점 | 가능한 영향 | 적용한 변경 | 검증 |
|---|---|---|---|
| 평문 또는 빠른 해시로 비밀번호 저장 가능 | DB 유출 시 비밀번호 즉시 노출·대입 공격 | 사용자별 난수 salt와 PBKDF2-SHA256 600,000회, 상수 시간 비교 | 해시가 매번 다르고 정상/오류 비밀번호 검증 테스트 |
| 인증 후에도 기존 익명 세션 재사용 | 세션 고정 공격 | 회원가입·로그인 성공 시 토큰 폐기 후 새 세션 발급 | 실제 브라우저 가입 후 인증 상태 확인 |
| 상태 변경 URL에 CSRF 보호 누락 가능 | 사용자 의사와 무관한 상품 삭제·구매 | 모든 POST 폼에 세션 CSRF 토큰, 서버 상수 시간 비교 | 토큰 없는 등록 요청 403 테스트 |
| 상품 ID만 알면 타인 상품 수정·삭제 가능 | 수평 권한 상승(IDOR) | 서버에서 `seller_id`와 현재 사용자 ID 비교 후 SQL에도 소유자 조건 추가 | 공격자 계정의 수정·삭제 403 테스트 |
| 문자열 결합 검색 쿼리 | SQL 삽입 및 데이터 노출 | 모든 SQL에 `?` 바인딩, LIKE 와일드카드 이스케이프 | `' OR 1=1--` 검색 테스트 |
| 상품명·설명·채팅을 그대로 HTML에 출력 | 저장형 XSS, 세션 탈취 | 모든 동적 출력 HTML escape, CSP의 script 기본 차단 | `<script>` 상품명 이스케이프 테스트 |
| 구매 상태 변경과 잔액 이전을 별도 수행 | 이중 구매, 포인트 생성·소실 | `BEGIN IMMEDIATE` 한 트랜잭션에서 잔액 조건부 차감→판매 상태 전환→판매자 적립→원장 기록 | 2회 구매 시 한 번만 송금되는 테스트 |
| 무제한 로그인 시도 | 계정 비밀번호 대입·아이디 변경을 통한 제한 우회 | IP+아이디 15분당 5회와 IP 전체 20회 제한을 함께 적용하고 사용자 존재 여부와 무관한 오류 메시지 사용 | 서로 다른 아이디 20개를 사용한 IP 제한 자동 테스트 |
| 존재하지 않는 계정만 빠르게 실패 | 응답 시간으로 가입 아이디 추측 | 실제 해시와 같은 600,000회 PBKDF2 더미 해시를 검증 | 실제·더미 해시 반복 횟수 자동 테스트 |
| 신고를 한 사용자가 반복 제출 | 임의 자동 차단 | `(reporter,target_type,target_id)` UNIQUE, 상품 3명/사용자 5명의 서로 다른 신고 필요 | 서로 다른 3명 신고 후 상품 숨김 테스트 |
| 관리자 URL만 숨기고 서버 권한 검사 누락 | 관리자 기능 탈취 | 모든 관리자 요청에 DB 세션의 `role=admin` 재검사 | 일반 사용자 `/admin` 403 테스트 |
| 로그에 IP·비밀번호 등 민감정보 기록 | 운영 로그 유출 | 비밀번호/메시지 본문 미기록, IP는 단방향 privacy hash로 저장 | 스키마·감사 함수 검토 |
| 관리자가 사용자 비밀번호를 직접 설정 | 관리자 오용·계정 탈취·행위 책임 부인 | 관리자에게 비밀번호 조회·임의 변경 권한을 부여하지 않음. 복구가 필요하면 일회용·단기 만료 재설정 토큰 방식 사용 | 관리자 라우트와 권한 테스트 검토 |
| 외부 콘텐츠·프레임 허용 | clickjacking·콘텐츠 주입 | CSP, `frame-ancestors 'none'`, `X-Frame-Options: DENY`, nosniff 등 | 응답 헤더 자동 테스트 |
| 확장자만 확인하거나 무제한 다중 사진 허용 | 실행 파일·대용량 이미지 저장, 메모리 고갈 | PNG/JPEG만 허용하고 확장자·MIME·시그니처 일치, 전체 디코딩, 픽셀 수를 검사한다. 최대 10장·파일당 8MB·요청당 32MB를 서버에서도 강제 | 정상 10장 저장·11장 거부, WebP·위장 이미지 거부 테스트 |
| 원본 사진을 그대로 공개 | EXIF 위치정보·숨은 데이터·파서 차이 노출 | 방향을 적용한 뒤 PNG/JPEG로 재인코딩하여 메타데이터와 후행 데이터를 제거하고 최대 변 길이를 4096px로 제한 | 저장 파일에서 테스트 메타데이터가 제거되는지 확인 |
| 차단 버튼만 만들고 서버에서 관계 미검사 | 차단 사용자가 채팅·구매를 계속 수행 | 양방향 차단 관계를 상품 목록·상세·구매·채팅에서 서버 측 검사 | 차단 후 상품 비노출, 채팅·구매 거부 테스트 |
| 관리자 제재 작업의 상태·대상 검증 부족 | 임의 신고 변경·관리자 계정 정지 | 관리자 역할, 신고/차단 ID 존재, 일반 계정 대상 여부를 매 요청 검사하고 감사로그 기록 | 신고 처리·차단 해제·계정 정지 통합 테스트 |
| 전체/임의 사용자 채팅 | 불필요한 사용자 접근·개인 메시지 노출 | 상품별 판매자-문의자 1:1 대화로 제한, 참가자와 상품 ID 동시 검사 | 제3자에게 대화 본문 미노출 테스트 |
| 신고 URL의 상품·사용자 ID 변조 | 대화하지 않은 사용자를 채팅 신고하거나 상품 맥락 위조 | 해당 상품에서 신고자와 피신고자가 실제 메시지를 주고받았는지 서버에서 확인하고, 검증된 상품 ID·제목을 서버가 신고 사유에 부착 | 실제 참여자의 신고·관리자 확인 성공, 제3자 신고 404 테스트 |
| 채팅 사진을 공개 정적 URL로 제공 | URL을 아는 제3자의 개인 사진 열람 | 상품 사진과 동일한 재인코딩을 적용하고, 인증된 대화 참가자만 전용 경로로 조회하도록 객체 권한 검사 | 참가자 조회 성공·제3자 및 공개 경로 조회 실패 테스트 |
| 복잡한 WebP 디코더까지 업로드 표면에 포함 | 파서 취약점·형식 혼동 공격 면적 증가 | 신규 업로드는 PNG/JPEG만 허용하고 WebP는 디코더 호출 전에 거부. 기존 참조 파일은 일회성 마이그레이션으로 PNG/JPEG 재인코딩 | WebP와 확장자 위장 WebP 거부 자동 테스트 |
| 인증 전 대형 multipart 요청 파싱 | 메모리·CPU 자원 고갈 | 업로드 경로를 허용 목록으로 제한하고 활성 로그인 사용자를 먼저 확인하며 전체 요청과 필드 크기를 제한 | 코드 검토와 업로드 경계 테스트 |
| 알림 읽음 상태를 클라이언트가 지정 | 타인의 메시지 읽음 처리·알림 조작 | 로그인 수신자와 현재 상품·상대방이 일치하는 메시지만 서버에서 읽음 처리 | 수신자 알림 1→대화 열람 후 0 테스트 |

## 운영 권고

1. `TINY_MARKET_ENV=production`과 HTTPS를 반드시 사용합니다.
2. DB와 백업 파일은 웹 루트 밖에 두고 최소 권한으로 접근합니다.
3. 리버스 프록시에서 전체 요청 속도 제한, 본문 크기 제한, TLS를 적용합니다.
4. 보안 이벤트와 오류율을 모니터링하고 감사 로그 보존·파기 기간을 정합니다.
5. Python 보안 업데이트를 적용하고 정기적으로 자동 테스트와 코드 검토를 반복합니다.
